# AGENTS.md

## Project
このリポジトリはブラウザで動くゲームプロトタイプ。Codexはゲーム仕様を勝手に変更せず、実装・検証・レポートを行う。

## Game design pillars
- プレイヤーが選択の因果関係を理解できること。
- プレイヤーが自律的に成長するアリのコロニーにうれしさを感じられること。
- プレイヤーがアリに感情移入し、敵に攻撃され、思うようにアリが成長できないことを悔しく感じること
- ランダム性は許可するが、結果の理由は説明可能にすること。
- 数値バランス変更は小さく、根拠を残すこと。
- 旧遠征モードは使わない。兵隊アリは平時は巣内にいて、プレイヤー指示で軍事タブから防衛出動または遠征出動すること。
- ライブ状態の味方アリは、警戒・逃走・戦闘中でも常にデフォルト色で描画すること。
- 敵襲の接近・開始・撃退は、軍事タブを開いていなくてもユーザーに分かる表示を保つこと。
- 今後の主体験は、同一マップ上の敵アリ巣との攻防と、敵巣の発見・陥落へ向かう進行に寄せること。
- 放置成長だけを主目的に戻さず、探索範囲、偵察、襲撃方向、敵巣攻撃の因果関係をプレイヤーが理解できる形にすること。

## Current gameplay rules
- 旧遠征モードは削除済み。通常アプリへ復活させない。遠征出動は同一マップ上の発見済み敵巣へ向かう地表出撃として扱い、旧 `src/expedition/` の抽象戦闘へ戻さない。
- 視点はプレイヤー巣固定にせず、広域マップを見回せるようにする。視点移動はセーブ状態ではなくランタイム操作として扱う。
- 兵隊アリは平時は巣内リソースとして扱い、プレイヤーが軍事タブから指示した時だけ地表へ出撃する。
- 働きアリ、土木アリ、兵隊アリ、特殊兵はすべて育房タブの育成キューを通し、種別ごとの食料コストと育成時間を経て1匹ずつ増える。強化購入だけで個体を直接増やさない。
- 育成キューは最大30件。アリ種ごとの育成上限は置かず、巣全体の収容上限と解禁条件だけで制限する。
- 出撃兵は見えている敵、または敵襲シグナル方向へ向かって前進する。巣周辺だけで待機させない。
- 敵アリ色は落ち着いた赤茶 `#8a4a2f` を維持する。
- 味方アリのライブ表示色は状態に関係なくデフォルト色を維持する。戦闘状態を色変更で表現しない。
- 戦闘は即ワープ/即ノックバックではなく、約2秒以上の組み合いとして見せる。
- 味方側は最大3匹まで1匹の敵に取り付き、正面・横・後ろ寄りから噛み付く配置を許可する。
- 戦闘後、敗者は画面内で固まらない。敵は画面外へ退却するか、死体として残る。死体は敵味方とも10秒で消える。
- 敵襲完了時の味方死亡数表示は、敵襲開始時の `fallenAnts` との差分を使う。`raid.casualties` 単体を表示の根拠にしない。
- 枝はアリが引っかかる原因になるため、通常マップの自然障害物として配置しない。
- 土木アリは工事に割り当てられている時だけ地表へ出る。未割当の土木アリは巣内リソースとして扱い、地表には描画しない。
- 土木工事は待機中の土木アリがいる限り、別種類の工事を同時に進められる。1つの工事へ無制限に密集させない。
- 土木工事の担当数は作業開始時1匹とし、各作業ごとに1〜10匹の範囲でプレイヤーが増減できる。
- 土木工事は `trailReinforce`、`lowBarricade`、`earthWall`、`sentryMound` を扱う。`lowBarricade` と `sentryMound` はボタン押下後に地面クリックで設置場所を指定する。`earthWall` は低い土塁より細く高い防壁として、プレイヤーがクリックで置いた頂点列から壁の位置と長さを指定し、敵の侵入を大きく遅らせ、壁上の味方攻撃を強める。
- `earthWall` の建築コストは、クリックした点を順につなぐ一筆書きの中心線の全長に応じて変わる。UIでは一筆線の指定中にも概算工数を表示し、完成時間は固定秒数として扱わない。
- `earthWall` の戦闘表現では、壁付近の味方は壁上に配置され、敵は壁を越える側で大きく減速する。低い土塁の既存効果を巻き添えで強化しない。
- `sentryMound` がない場合、敵襲予兆段階では敵襲方角を表示しない。方角アラームや予兆中の兵隊ターゲットで隠し方角を漏らさない。
- 完成済み `sentryMound` がある場合、敵襲予兆段階で方角を捕捉し、警告時間を少し延ばす。プレイヤーが兵隊を出撃すると、その方角に対して前方布陣を取る。
- 各土木工事は `buildCost` と所要目安を持ち、UIでは固定秒数ではなく工数と「採土・往復・担当数で変動」する説明を表示する。
- 土木工事の時間説明は、採土・往復・担当数で変動することを伝える。内部progress量だけを「約n秒」のように実時間として表示しない。
- 巣は盛り上がった蓋状のドームではなく、地面に開いた小さな穴として描画する。入口穴の見た目は大きくしすぎない。穴に入った味方アリは地表に表示せず、最低10秒は外へ出さない。
- 一度の出撃指示で出せる兵隊アリは、通常兵隊アリ、重兵装アリ、盾頭アリ、酸射アリ、斥候アリ、救護アリ、小隊長アリを合算した総数の半分までにする。端数は切り上げ、総数1匹の時は1匹出撃可能にする。これは1回あたりの出撃上限であり、クールダウン後は巣内に残っている兵隊を追加出撃できる。
- 重兵装アリも通常兵隊アリと同じく巣内兵力として扱い、平時は地表に出さない。プレイヤーが軍事タブから出撃指示した時だけ地表に表示する。
- 盾頭アリも巣内兵力として扱い、平時は地表に出さない。出撃時は前線に出て味方の前でタンク役として構え、敵とは組み合い戦闘を発生させず、味方を守る壁として敵を押し返す。
- 斥候アリも巣内兵力として扱い、平時は地表に出さない。出撃時は敵を標識して味方の狙いを揃える支援役とし、敵とは組み合い戦闘を発生させない。
- 救護アリも巣内兵力として扱い、平時は地表に出さない。出撃時は消耗・負傷した味方を支援して後方へ退避させる支援役とし、敵とは組み合い戦闘を発生させない。
- 小隊長アリも巣内兵力として扱い、平時は地表に出さない。出撃時だけ一時小隊を編成し、味方の集合位置と集中目標を揃える。小隊編成は現状セーブしないランタイム状態とする。
- 複数小隊が同時に出撃する場合、兵科は小隊間でなるべく均等に配分し、敵が複数いる時は全小隊が同じ目標へ集中しすぎないよう目標と進路を分散する。
- 敵襲の敵数は巣の規模に合わせて増える。大規模な巣でも敵が少数に固定されて危機感が消えないようにする。
- マップはプレイヤー巣周辺の円形初期視界から始まり、領土拡大、見張り塚、斥候によって視界が広がる。未踏破領域は薄い霧ではなく暗闇として強く覆い、霧の外の敵や敵巣を直接ターゲットさせない。
- アリが地表で入った領域はランタイム中の探索済み領域として霧が晴れ、敵巣や敵の視認にも使う。現時点ではセーブ対象にしない。
- 自然餌場は食べ尽くされた直後に即復活せず、一定時間後に元の餌場近辺へ再生成する。
- 近距離の自然餌場は小規模コロニーを支える量に留め、大規模コロニーは中距離・遠距離の餌場へ向かう必要がある。遠距離餌は量が多いが配送効率が落ち、敵襲時に救援が届きにくいリスクを持つ。
- トップバーの採餌表示は推定 `foodRate` ではなく、直近60秒にアリが実際に持ち帰った食料を毎分換算した「直近採餌/分」とする。オフライン進行用の `foodRate` と混同しない。
- 採餌系の成長は抽象的な「採餌力」表示を伸ばすのではなく、働きアリの運搬量、採餌中の移動速度、遠方採餌の往復効率など、ライブ挙動に見えるパラメーターへ反映する。
- Lv3以降の巣拡張は、累計食料と個体数だけでなく、遠方採餌で伸びる領土も必要にする。遠方採餌の進捗端数は現時点ではランタイム扱いで、獲得済みの整数領土だけを既存セーブ項目に保存する。
- 敵アリの巣は同一マップ上の遠方に置く。未発見時も襲撃の発生源として扱うが、敵巣位置や襲撃方角は見張り塚・斥候・発見済み敵巣のいずれかがある時だけ表示する。
- 軍事タブでは防衛出動と遠征出動を選べる。防衛出動は敵襲・見えている敵への対応を優先し、遠征出動は発見済みの敵巣を目標にする。最終的な主目的は敵巣を陥落させることとする。

## Allowed changes
- Playwright設定、テスト、検証スクリプト、data-testid追加、開発時限定debug hook。
- 既存挙動を保つための小さな修正。
- docs/decision-log.mdへの追記。
- 今後のCodex引き継ぎに必要なルール、検証手順、運用上の注意が増えた場合のAGENTS.md更新。

## Architecture notes
- `src/simulation.ts` はまだ通常ゲームの compatibility entrypoint / orchestration 層。新しい定義や拡張点を無条件にここへ増やさず、既存 registry/state 境界を優先する。
- アリ種別の追加・調整は `src/config/variants.ts` を起点にする。既存アリの `id`, `position`, `heading`, `gaitPhase`, `renderIndex` の連続性を壊さない。救護アリのような兵科追加では `src/state/colony.ts` の save version、`src/state/migrations.ts`、save/load test も合わせて更新する。
- 育房の種別ごとの育成コスト、育成時間、解禁条件は `src/config/barracks.ts` を起点にする。
- 土木工事の種類追加・調整は `src/config/construction.ts` の `CONSTRUCTION_KINDS` / `CONSTRUCTION_DEFS` を起点にする。`src/config/balance.ts` に土木カタログを戻さない。
- 土木工事の見た目追加・調整は `src/render/VoxelBuildingRenderer.ts` を優先する。保存形式と効果計算は `EarthworkState` の中心/半径/回転モデルを維持し、描画都合でボクセル単位のセーブへ広げない。
- `EarthworkKind` は `ConstructionKind` に追従する。土木種別を追加する場合は、`src/state/migrations.ts` の正規化、保存互換、unit test、`npm run eval:save` を必ず確認する。
- colony state、derived計算、save/migration は `src/state/*` を優先して扱う。セーブキー `ant3d.colonyState` は不用意に変えない。
- `src/expedition/` は旧遠征/検証由来。通常アプリへ旧遠征モードを戻す根拠にしない。遠征出動は通常マップ上の兵隊出撃ロジックで扱う。

## Forbidden changes
- ゲーム仕様の暗黙変更。
- UI文言の大幅変更。
- 本番buildへのdebug hook露出。
- セーブデータ互換性を壊す変更。
- unrelated refactor。
- テストを通すためだけのゲームロジック改変。

## Commands
- Install: `npm install`
- Dev: `npm run dev`
- Unit test: `npm run test`
- Smoke eval: `npm run eval:smoke`
- Save/load eval: `npm run eval:save`
- Combat visual eval: `npm run verify:combat`
- Balance eval: `npm run verify:balance`
- Raid quick check URL: `/?raidSoon=1`
- Playwright eval server の既定TTLは15分。大きな敵襲規模を含むmobile/desktop smokeが途中で `ERR_CONNECTION_REFUSED` にならないよう維持する。

## Deployment
- GitHub Pages は `main` push で `.github/workflows/deploy.yml` が自動デプロイする。
- main 以外のブランチを一時デプロイする場合は、`github-pages` environment の branch policy を一時追加し、デプロイ後に必ず削除する。
- main への push はユーザーが明示した場合だけ実行する。
- ユーザーが実装後のデプロイまで求めている流れでは、明示的な停止指示がない限り、検証後に現在ブランチをpushして一時branch policy経由でデプロイまで進める。

## Maintenance
- 作業中に恒久的な注意点、禁止事項、検証手順、リポジトリ運用ルールが増えた場合は、必要に応じてAGENTS.mdを更新する。
- 一時的な作業メモや今回限りの結果はAGENTS.mdではなく、最終報告またはdocs/decision-log.mdに残す。

## Definition of Done
- `npm run test` が通る。
- `npm run eval:smoke` が通る。
- セーブ機能がある場合は `npm run eval:save` が通る。
- 戦闘描画や敵襲AIを変更した場合は `npm run verify:combat` を実行し、出力スクリーンショットを確認する。
- 兵種・敵襲・防衛バランスを変更した場合は `npm run verify:balance` を実行し、`verification/balance/summary.json` の失敗がないことを確認する。
- 失敗時はPlaywright report、screenshot、console error、原因候補を報告する。
- 最終報告に変更点、検証結果、残リスク、未対応事項を含める。

## Stop conditions
- 同じエラーで3回連続失敗。
- 仕様判断が必要。
- セーブ仕様変更が必要。
- production露出やデータ破壊の可能性がある。
